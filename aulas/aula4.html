Visibilidade, Modificadores de Acesso e Encapsulamento

Aula 4: Unidade II – Noções básicas e de OO e seus conceitos

Ementa e Contextualização

Até agora, nossos atributos e métodos estavam "abertos" para o mundo. Mas e se quiséssemos proteger os dados de um objeto contra modificações indevidas? É aqui que entram os modificadores de acesso e o encapsulamento. O encapsulamento é um dos pilares da POO e consiste em "esconder" os detalhes internos de um objeto, expondo apenas uma interface segura para interação. Isso não só aumenta a segurança do software, mas também o torna mais flexível e fácil de manter.

🛡️ Protegendo Seus Objetos

1. Modificadores de Acesso

Java oferece quatro níveis de controle de acesso para membros de uma classe (atributos e métodos):

  • publicAcessível de qualquer classe, em qualquer pacote. É o nível menos restritivo.
  • protectedAcessível dentro do mesmo pacote e por subclasses (mesmo em pacotes diferentes).
  • package-private(Sem modificador) Acessível apenas por classes dentro do mesmo pacote. É o padrão.
  • privateAcessível apenas dentro da própria classe. É o nível mais restritivo.

2. Encapsulamento: O Princípio da Caixa-Preta

Encapsular significa agrupar os dados (atributos) e os métodos que operam nesses dados dentro de uma única unidade (a classe), e esconder os detalhes da implementação do mundo exterior. A boa prática é declarar os atributos como private e fornecer métodos public para acessá-los e modificá-los de forma controlada.

Pense em um caixa eletrônico: você não tem acesso direto ao dinheiro no cofre (atributo private). Em vez disso, você usa uma interface pública (a tela e os botões) para realizar operações seguras como sacar() e verificarSaldo().

💻 Exemplos Práticos: Gerenciando uma Conta Bancária

Exemplo 1: Classe ContaBancaria Encapsulada

Note como o saldo é private. A única forma de alterá-lo é através dos métodos públicos depositar e sacar. 


public class ContaBancaria {
    // Attributes are private to protect them from direct external access.
    private String titular;
    private double saldo;

    // A public constructor to initialize the object.
    public ContaBancaria(String titular) {
        this.titular = titular;
        this.saldo = 0.0; // Initial balance is always zero.
    }

    // Public method to deposit money. This is a controlled way to modify 'saldo'.
    public void depositar(double valor) {
        if (valor > 0) {
            this.saldo += valor;
            System.out.println("Depósito de R$ " + valor + " realizado com sucesso.");
        } else {
            System.out.println("Valor de depósito inválido.");
        }
    }

    // Public method to withdraw money.
    public void sacar(double valor) {
        if (valor > 0 && valor <= this.saldo) {
            this.saldo -= valor;
            System.out.println("Saque de R$ " + valor + " realizado com sucesso.");
        } else {
            System.out.println("Saque não permitido. Saldo insuficiente ou valor inválido.");
        }
    }

    // Public "getter" method to allow read-only access to the balance.
    public double getSaldo() {
        return this.saldo;
    }
}

Exemplo 2: Usando a Classe Encapsulada

Na classe Banco, não podemos fazer minhaConta.saldo = -500;. Somos forçados a usar os métodos seguros. 


public class Banco {
    public static void main(String[] args) {
        ContaBancaria minhaConta = new ContaBancaria("Ana Souza");

        System.out.println("Saldo inicial: R$ " + minhaConta.getSaldo());

        minhaConta.depositar(500.0);
        minhaConta.sacar(200.0);
        
        // This line would cause a compilation error because 'saldo' is private.
        // minhaConta.saldo = 1000000; 
        
        minhaConta.sacar(400.0); // This will fail due to insufficient funds.

        System.out.println("Saldo final: R$ " + minhaConta.getSaldo());
    }
}

Atividade Prática (TED): Refatorando a Classe Funcionario

Vamos refatorar uma classe Funcionario para aplicar o encapsulamento e garantir a integridade dos seus dados.

  1. Crie a classe Funcionario.java com os atributos private String nome e private double salario.
  2. Implemente os métodos public getters e setters para ambos os atributos: getNome(), setNome(String nome), getSalario(), setSalario(double salario).
  3. Adicione uma regra de negócio no setSalario: o salário não pode ser definido com um valor menor que o salário mínimo (R$ 1.212,00). Se o valor for inválido, exiba uma mensagem de erro e não altere o salário.
  4. Crie uma classe Empresa.java com um método main para testar. Crie um objeto Funcionario, tente definir um salário válido e um inválido, e exiba os resultados usando os getters.

Sugestão de Metodologia Ativa

Realize esta atividade em dupla (*pair programming*). Discutam por que a validação no método setSalario é importante para a consistência do sistema. Como essa abordagem ajuda a evitar bugs no futuro?
